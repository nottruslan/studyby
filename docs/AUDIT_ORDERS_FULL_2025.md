# Полный аудит: Биржа заказов и Админка

**Дата:** март 2025  
**Разделы:** Биржа заказов (студент), Админка (модератор).

---

## 1. Биржа заказов (студент) — что есть и чего не хватает

### 1.1 Критичные пробелы (ломают сценарий)

| Проблема | Сейчас | Нужно |
|----------|--------|--------|
| **Клик по заказу → 404** | Ссылки ведут на `/orders/[id]`, маршрута нет. | Страница **`/orders/[id]`**: просмотр заказа (все поля, файлы, статус, цена). |
| **Нет действий «Одобрить цену» / «Отклонить»** | При статусе «Ожидает оплаты» только показ цены в списке. | На странице заказа кнопки: **«Одобрить цену»** (→ `in_progress`), **«Отклонить»** (→ `cancelled`). |
| **Файлы заказа не видны и не скачиваются** | В списке и при переходе по заказу файлов нет. | На странице заказа: список файлов + **«Скачать»** (signed URL для своего заказа). |
| **Ошибка загрузки списка не видна** | При ошибке Supabase список пустой, без сообщения. | Отдельный empty state «Не удалось загрузить заказы» + кнопка **«Повторить»** и/или toast. |

### 1.2 Редактирование и отмена (студент)

| Действие | Сейчас | Рекомендация |
|----------|--------|--------------|
| **Редактировать заказ** | Нет. После создания изменить название, срок, описание, файлы нельзя. | Для статусов `review` (и при желании `waiting_payment`) — кнопка **«Редактировать»** на странице заказа и форма/модалка редактирования (или отдельная страница `/orders/[id]/edit`). |
| **Отменить заказ** | Нет. Удалить или отменить заказ нельзя. | Для `review` и `waiting_payment`: кнопка **«Отменить заказ»** (→ `cancelled`) на странице заказа. |
| **Удалить (полностью)** | Нет. | Обычно достаточно отмены (статус `cancelled`). Жёсткое удаление из БД — опционально, только для админа. |

### 1.3 UX списка и формы

| Пункт | Статус | Рекомендация |
|-------|--------|--------------|
| Вкладки «Активные» / «История» | Ок | — |
| Бейджи статусов, цена при `waiting_payment` | Ок | — |
| Пустые состояния | Ок | — |
| Подтверждение при уходе с формы создания | Нет | При «Назад» с заполненной формой — диалог «Данные не сохранятся. Выйти?» или кнопка «Отмена». |
| Текст кнопки при отправке | Только спиннер | Оставить текст «Создание…» рядом со спиннером. |
| Блокировка шагов при submit | Нет | На время отправки отключать «Назад»/«Далее» или блокировать уход. |
| Лимит описания в UI | Нет | Счётчик «N / 2000» у textarea или `maxLength`. |
| Даты в МСК | В списке локальная зона | На странице заказа и в списке — `toLocaleString('ru-RU', { timeZone: 'Europe/Moscow' })`. |
| Просроченный дедлайн | Не выделяется | На карточке/странице заказа — бейдж «Просрочено» или цвет. |

### 1.4 Real-time и прочее

- **Real-time:** список загружается один раз; смена цены/статуса админом не видна без перезагрузки. Желательно: подписка на изменения своих заказов (Supabase Realtime) с обновлением списка и страницы заказа.
- **Лимиты файлов:** в форме создания нет ограничений по количеству и размеру — стоит ввести (например 5 файлов, 10 MB на файл) и показывать в UI/ошибке.

## 2. Админка — что есть и чего не хватает

### 2.1 Уже сделано

- Маршрут `/admin/*` защищён по `profiles.role === 'admin'` в middleware.
- Страница `/admin/orders`: таблица (десктоп) и карточки (мобилка), фильтр по статусу, сортировка по дате.
- Колонки: ID, Студент, Предмет, Тип, Дедлайн, Статус, Цена, Действия.
- Кнопка **«Редактировать»** → Sheet с полной информацией, ссылками на файлы (signed URL), полями «Статус» и «Цена», кнопка **«Сохранить»** → Server Action `updateOrderAdmin`.
- Отдельный layout админки, сайдбар (десктоп), гамбургер + Sheet (мобилка).

### 2.2 Чего не хватает в админке

| Пункт | Описание |
|-------|----------|
| **Удаление заказа** | Нет кнопки «Удалить» (или «Отменить» с переводом в `cancelled`). Если нужно именно удаление из БД — отдельная Server Action только для админа. |
| **Просмотр без редактирования** | Сейчас только «Редактировать» (Sheet). Можно добавить «Открыть» / клик по строке для режима только просмотра (тот же Sheet без полей редактирования или отдельная страница `/admin/orders/[id]`). |
| **Обратная связь после сохранения** | Есть toast и `router.refresh()`. Можно при желании явно обновлять строку в таблице без полной перезагрузки. |
| **Real-time** | При работе нескольких модераторов изменения не подтягиваются автоматически — по желанию подписка на таблицу заказов. |
| **Поиск / фильтры** | Только фильтр по статусу. При росте числа заказов — поиск по названию/студенту/предмету. |

| **P0** | На странице заказа при статусе `waiting_payment`: кнопки **«Одобрить цену»** и **«Отклонить»** (Server Action от имени студента, смена статуса). |

## 4. Рекомендуемый порядок внедрения

1. **Страница заказа `/orders/[id]`**  
   - Server Component или client fetch по `id` (проверка `student_id === user.id`).  
   - Вывод: название, предмет, тип, дедлайн, описание, статус, цена (если есть), список файлов с получением signed URL для своего заказа (Server Action или API).

2. **Действия студента на странице заказа**  
   - Server Actions: `approveOrderPrice(orderId)` (→ `in_progress`), `rejectOrder(orderId)` (→ `cancelled`).  
   - Показывать кнопки только при `status === 'waiting_payment'`.

3. **Ошибка загрузки списка заказов**  
   - В `orders/page.tsx`: при `error` от Supabase не только `setOrders([])`, но состояние `loadError: true`, отдельный блок с текстом и кнопкой «Повторить».

4. **Редактирование и отмена заказа студентом**  
   - На странице заказа для `review` (и при необходимости `waiting_payment`): кнопки «Редактировать» (форма/модалка или `/orders/[id]/edit`) и «Отменить заказ» (Server Action → `cancelled`).

5. **Админка: отмена/удаление**  
   - В Sheet или в таблице: кнопка «Отменить заказ» (статус `cancelled`) и при необходимости «Удалить» (DELETE через admin client).

После этого раздел биржи заказов и админка будут доведены до полноценной работы с точки зрения UX и логики.
