# Аудит раздела «Биржа заказов»

## 1. Критические пробелы

### 1.1 Нет страницы заказа `/orders/[id]`
- В списке все карточки ведут на `href={/orders/${order.id}}`, но маршрута **нет** → клик даёт **404**.
- **Нужно:** добавить `app/(main)/orders/[id]/page.tsx`: просмотр заказа (все поля, файлы), для статуса `waiting_payment` — кнопки «Одобрить цену» и «Отклонить».

### 1.2 Нет действий по заказу в статусе «Ожидает оплаты»
- По ТЗ при `waiting_payment` должны быть: **цена** + кнопки «Одобрить цену» (→ `in_progress`) и «Отклонить» (→ `cancelled`).
- Сейчас на списке только показывается цена, кнопок нет. Логично разместить их на странице заказа `[id]`; при желании можно дублировать на карточке в списке.

### 1.3 Модератор не может выставить цену
- Схема: студент создаёт → модератор вручную назначает цену и переводит в `waiting_payment`.
- В БД RLS: только владелец (`student_id = auth.uid()`) может читать/писать свои заказы. Роли «модератор» и политик для неё **нет**.
- **Нужно:** либо роль в `profiles` (например `role: 'student' | 'moderator'`) + RLS-политика «модератор может обновлять `price` и `status` у любых заказов», либо отдельный бэкенд/Edge Function с service role для действий модератора.

## 2. Логика и UX

### 2.1 Real-time по ТЗ не реализован
- ТЗ: «Use supabase.channel to update price and status instantly without refresh».
- Сейчас список загружается один раз в `useEffect`; при смене цены/статуса модератором пользователь не видит изменений без перезагрузки.
- **Нужно:** подписка на `supabase.channel('orders')` с фильтром по `student_id` (или по `id` заказа на странице заказа) и обновление локального state при `UPDATE`.

### 2.2 Формат даты
- ТЗ: «Format all dates as "25 мая, 18:00 (МСК)"». Сейчас `formatDate()` даёт именно такой формат, но время в локальной зоне браузера, а не явно МСК. Для строгого «(МСК)» нужно конвертировать в московское время (например через `toLocaleString('ru-RU', { timeZone: 'Europe/Moscow' })` или библиотеку дат).

### 2.3 Дедлайн в прошлом
- В форме создания дата ограничена `min={getMinDatetimeLocal()}` — прошедшие даты выбрать нельзя, ок.
- На списке/карточке заказа просроченный дедлайн не выделяется (можно добавить визуально «Просрочено» или цвет).

### 2.4 Файлы заказа
- В БД хранятся пути в Storage (`order-files/{orderId}/{filename}`). Бакет приватный — для скачивания нужны signed URL.
- На списке и на странице заказа файлы не показываются и не скачиваются. На странице `[id]` нужно: список файлов + кнопка «Скачать» (получение signed URL через `supabase.storage.from('order-files').createSignedUrl(path)`).

## 3. Форма создания заказа (`/orders/new`)

### 3.1 Сделано хорошо
- Многошаговая форма (Предмет/тип → Срок → Описание и файлы), валидация Zod по шагам и при отправке.
- Срок: `datetime-local` с `min` — прошедшие даты недоступны.
- Загрузка файлов в бакет `order-files`, пути пишутся в `orders.files`.
- Тосты (sonner) при успехе/ошибке.

### 3.2 Небольшие замечания
- **Deadline по ТЗ:** «Deadline via Popover+Calendar» — сейчас обычный `input datetime-local`. Для соответствия ТЗ можно позже заменить на Popover + календарь (например Shadcn Calendar).
- **Drag-and-drop:** в интерфейсе написано «Перетащите файлы сюда», но обработчика `onDrop` на label нет — только выбор через клик. Либо добавить `onDrop`/`onDragOver`, либо убрать формулировку про перетаскивание.
- **Лимит файлов/размера:** нет ограничения по количеству файлов и по размеру; при большом объёме возможны ошибки Storage/таймауты. Имеет смысл ограничить (например 5 файлов, 10 MB на файл) и показывать ошибку в форме.

## 4. Список заказов (`/orders`)

### 4.1 Сделано хорошо
- Вкладки «Активные» / «История», пустые состояния с иконками и CTA «Создать заказ».
- Бейджи статусов по ТЗ: «На проверке» (жёлтый), «В работе» (индиго), плюс «Ожидает оплаты», «Выполнен», «Отменён».
- Для `waiting_payment` выводится цена.
- Даты в формате «день месяц, часы:минуты (МСК)» (с оговоркой про часовой пояс из п. 2.2).

### 4.2 Замечания
- После создания заказа делается `router.refresh()` — список обновится при переходе на `/orders`, но без real-time новые заказы от других вкладок/устройств не появятся (см. п. 2.1).
- Нет индикации «заказ на проверке» (например подзаголовок «Модератор назначит цену в течение …»).

## 5. Безопасность и данные

### 5.1 RLS
- Студент видит и меняет только свои заказы — корректно.
- Модератору для назначения цены нужен либо отдельный путь с повышенными правами (см. п. 1.3), либо серверный API.

### 5.2 Storage `order-files`
- Политики: авторизованные пользователи могут загружать и читать. Имеет смысл уточнить: студент может читать только файлы своих заказов (по префиксу пути `orderId`, где `orderId` — id заказа с `student_id = auth.uid()`). Сейчас политика «Authenticated can read order files» даёт доступ ко всем файлам бакета любому авторизованному пользователю.
