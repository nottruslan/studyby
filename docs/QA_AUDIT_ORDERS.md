# Аудит раздела «Биржа заказов» (QA & Product)

**Роль:** Senior QA Engineer + Product Manager  
**Объект:** раздел Заказы (список, форма создания), UI, логика, БД.  
**Код не менялся — только анализ и отчёт.**

---

## 1. UX И НАВИГАЦИЯ (UX/UI Dead Ends)

### Навигация
- **Форма «Новый заказ»:** есть кнопка «Назад» (ведёт на `/orders`) — выход есть. Явной кнопки **«Отмена»** нет: пользователь может не понять, что «Назад» отменяет ввод (данные при возврате теряются без предупреждения).
- **Список заказов:** с списка уйти можно только через сайдбар/таббар — ок. Кнопки «Назад» на странице списка нет (не обязательна, т.к. это не форма).
- **Страница заказа `/orders/[id]`:** маршрута **нет**. Клик по карточке ведёт в **404** — тупик. Нет ни просмотра заказа, ни модалок с деталями.

### Empty states
- **Список:** при пустых «Активные» и «История» показываются заглушки с иконкой, текстом и кнопкой «Создать заказ» — хорошо.
- **Нет отдельного empty state для ошибки загрузки:** при ошибке запроса к БД список просто пустой (`setOrders([])`), пользователь видит ту же заглушку «Пока нет активных заказов» и не понимает, что произошла ошибка (сеть, RLS, и т.д.).

### Текущее состояние (Loading, disabled)
- **Список:** при первой загрузке показывается спиннер (Loader2) — ок. Skeleton UI нет.
- **Форма отправки:** кнопка «Создать заказ» в шаге 3 имеет `disabled={loading}` и при loading показывает только спиннер (текст кнопки пропадает) — пользователь не видит «Создаётся…», только иконку. Остальные кнопки («Назад», «Далее») во время отправки не блокируются: можно нажать «Назад» и уйти со страницы во время запроса.
- **Список после создания заказа:** обновление через `router.refresh()`; явного повторного запроса или индикатора «обновлено» нет — в целом ок, но при медленной сети может казаться, что заказа нет.

## 2. БИЗНЕС-ЛОГИКА И CRUD

### Create (создание)
- Реализовано: форма, валидация Zod, вставка в `orders`, загрузка файлов в Storage, обновление `files`. RLS: только свой `student_id`.

### Read (чтение)
- Реализовано: список своих заказов по `student_id`. Детальной страницы заказа **нет** — просмотр одной записи (описание, файлы, даты, статус) недоступен.

### Update (редактирование)
- **В UI нет:** нельзя отредактировать название, предмет, срок, описание или файлы после создания. В БД RLS разрешает пользователю обновлять свои заказы (`student_id = auth.uid()`), но интерфейса для этого нет.
- **По продукту:** для статуса `waiting_payment` нужны действия «Одобрить цену» (→ `in_progress`) и «Отклонить» (→ `cancelled`) — ни кнопок, ни страницы, где их разместить, нет.

### Delete (удаление)
- **В UI нет:** удалить заказ нельзя. В БД политики на DELETE не проверялись отдельно; при наличии RLS «только свой заказ» удаление своего заказа через API возможно, но в приложении не предлагается. Нет ни «Отменить заказ», ни «Удалить» для подходящих статусов.

### Права доступа
- Список и запись идут через Supabase client с сессией пользователя; выборка по `student_id = auth.uid()`. Чужие заказы студент не видит и не меняет — ок. Роли модератора нет: выставить цену и перевести в `waiting_payment` через текущий UI/RLS нельзя.

## 3. КРАЙНИЕ СЛУЧАИ И ОБРАБОТКА ОШИБОК

### Длина ввода
- **Название:** Zod `max(200)` — при 10 000 символах валидация не пройдёт, покажется ошибка и toast «Проверьте поля формы».
- **Описание:** `max(2000)` — аналогично. В textarea нет `maxLength`, пользователь может набирать до отправки и только тогда увидеть ошибку (лучше бы лимит в UI + сообщение «осталось N символов» или подсветка при превышении).

### Сеть и ошибки
- **Пропажа интернета при отправке формы:** `createErr` или исключение при upload попадают в `catch`, показывается `toast.error(msg)`. Текст берётся из `err.message` или «Ошибка создания заказа». Повторить отправку можно (кнопка снова активна после `setLoading(false)`), но введённые данные не восстанавливаются из «черновика» — нормально для текущего уровня.
- **Ошибка загрузки списка:** при `error` от Supabase список обнуляется, **toast не показывается**, пользователь видит empty state как при пустом списке — некорректно с точки зрения обратной связи.

### Двойной клик (Double Submit)
- Кнопка «Создать заказ» при `loading` имеет `disabled={loading}` — второй клик не отправляет запрос. Но между кликом и установкой `setLoading(true)` есть микро-задержка; теоретически два быстрых клика могут дать два вызова `handleSubmit`. Надёжнее дополнительно блокировать повторную отправку (флаг или отключение кнопки в начале `handleSubmit` до `setLoading(true)`).

### Другие краевые случаи
- **Сессия истекла во время заполнения формы:** при submit получим ошибку от Supabase; показывается toast, редиректа на логин нет (есть только при `!user` → «Войдите в аккаунт» и `router.push("/onboarding")`).
- **Очень большие или много файлов:** лимитов в коде нет; возможны таймаут, ошибка Storage или долгая загрузка без прогресса. Пользователю не объяснено, сколько файлов и какого размера можно прикрепить.

## 4. PWA / МОБИЛЬНЫЕ ОГРАНИЧЕНИЯ

### Размер зон нажатия (Touch targets)
- Рекомендация (Apple HIG / Material): не менее **44×44 pt (px)**.
- **Таббар:** кнопки с `py-2 px-4`, иконка h-6 w-6 + текст — высота области порядка 36–40 px, может быть чуть меньше 44.
- **«Создать заказ» (список):** `h-9` (36 px) — ниже 44.
- **Вкладки «Активные» / «История»:** `px-4 py-2` — высота тоже около 36 px.
- **Кнопки в форме** (Назад, Далее, Создать заказ): стандартная высота кнопок (h-10 и т.д.) обычно 40 px — ближе к норме, но не везде 44.
- **Крестик удаления файла:** только `p-1` и иконка h-4 w-4 — зона нажатия очень маленькая, на мобильном неудобно.

### Нижний таббар и контент
- У `main-app-shell` задан `pb-[calc(5rem+env(safe-area-inset-bottom))]`, у контента есть отступ снизу — таббар не перекрывает основной контент. Кнопка «Создать заказ» в empty state и последние карточки списка не уходят под таббар. В standalone свои отступы через `data-display-mode="standalone"` — учтено.

- **Позже:** редактирование/отмена заказа, лимиты файлов, drag-and-drop, единый минимум 44px для touch, real-time.
